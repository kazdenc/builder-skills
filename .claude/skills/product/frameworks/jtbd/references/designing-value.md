# Designing Value

You know the job. You've scored the needs. Now design the right thing — and make sure you're not just guessing.

## Roadmaps Built on Jobs

Stop organizing roadmaps by feature. Organize them by the needs you're going after.

### How to Build One

1. **Start with the job map.** Lay out the stages.
2. **Overlay your scored needs.** Place the biggest gaps at each stage.
3. **Sequence by opportunity score.** Highest-scoring unmet needs first.
4. **Group into themes.** Cluster related needs into initiatives.

```
Phase 1 → [Stage with biggest unmet needs]
           Tackle need cluster A (score: 16)
           Tackle need cluster B (score: 14)

Phase 2 → [Next highest-opportunity stage]
           Tackle need cluster C (score: 13)

Phase 3 → [Expand into adjacent stages or related jobs]
           Start addressing emotional/social layer
```

### Why This Works Better

- **Stable.** Jobs and needs shift slowly. Your roadmap won't need constant overhaul because a competitor shipped something.
- **Shared language.** "Minimize the time to gather documents" makes sense to engineering, design, sales, and leadership. Feature names don't travel as well.
- **Defensible.** When someone asks "why are we building this?", you point to scored data from real performers, not an internal hunch.

## Job Stories

Replace user stories. The "As a [persona]" format anchors to who someone is. Job stories anchor to the situation they're in — which is what actually drives behavior.

### Format

```
When [situation],
I want to [motivation],
so I can [outcome].
```

### Side by Side

| User Story | Job Story |
|-----------|-----------|
| As a user, I want to search restaurants so I can find somewhere to eat | When I'm in an unfamiliar area and hungry, I want to find nearby food, so I can minimize the time between deciding to eat and sitting down |
| As an admin, I want to export data so I can share reports | When stakeholders need a progress update, I want to pull key metrics together, so I can maximize the chance that the next decision is informed by real numbers |

### What Makes a Good One

- **Rooted in research.** Every story should trace back to something a performer said or a need you scored. If you can't point to the data, you're guessing.
- **Situation-first.** The "when" is the most important part. It's the trigger, the context, the thing that makes this need acute right now.
- **Outcome maps to a need.** The "so I can" should match your need statement format — a measurable outcome, not a vague benefit.
- **No solution prescribed.** The story frames the problem. The team figures out the solution.

## Structuring the Solution

Use the job map as a blueprint. Each stage of the job is a potential module, feature area, or capability.

### Mapping Solutions to the Job

| Job Stage | Key Unmet Need | What the Solution Could Do |
|-----------|---------------|---------------------------|
| Define | Takes too long to figure out what's needed | Templates, smart defaults, guided setup |
| Locate | Hard to find the right inputs | Search, recommendations, integrations |
| Prepare | Setup is error-prone | Validation, pre-checks, auto-configuration |
| Execute | Too many manual steps | Automation, real-time guidance |
| Monitor | Can't tell if it's going well | Dashboards, alerts, progress indicators |
| Finish | Wrapping up is tedious | Auto-summaries, one-click export |

### How Much of the Job to Own

Three levels, each with tradeoffs:

- **Point solution** — Own one or two stages. Faster to build, easier to sell, lower ceiling. Good for starting.
- **Platform** — Own the full job end-to-end. Harder to build, higher switching costs for users, bigger long-term opportunity.
- **Ecosystem** — Extend into related jobs and other performer roles. Highest complexity, market-defining potential.

**Start narrow, expand along the job map.** Each new stage you serve makes it harder for competitors to displace you and easier for performers to stay.

## Testing Your Bets

Every product decision rests on assumptions. Make them explicit, then test the riskiest ones first.

### Name Your Assumptions

For each thing you're planning to build, write down what has to be true for it to work:

| Assumption | What could go wrong | How to test |
|-----------|-------------------|-------------|
| Performers actually spend 3+ hours on this stage | Maybe it's only painful for a few | Time diary study, interview follow-up |
| This need is the top unmet one | Maybe we misread the scores | Quantitative survey, 50+ respondents |
| Our approach cuts the time in half | Maybe the real bottleneck is elsewhere | Prototype test with 5-8 performers |
| People will switch from their current workaround | Maybe habit is too strong | Smoke test, landing page, sign-up rate |

### Test in This Order

1. **Is the need real?** (If not, nothing else matters)
2. **Will people switch?** (Real need doesn't guarantee adoption — four forces apply)
3. **Can we build it?** (Feasibility)
4. **Can we sustain it?** (Unit economics, operations)

### Before Launch, Check the Forces

Even if the solution works, adoption depends on whether the forces favor switching:

- **Is the push strong enough?** The performer's current pain has to be acute, not just annoying.
- **Is the pull compelling?** They need to see clearly that your thing gets the job done better.
- **Have you lowered anxiety?** Free trial, money-back guarantee, migration support, social proof.
- **Have you broken the habit?** Import tools, familiar patterns, zero-configuration start.

If the opposing forces outweigh the generating ones, the product will stall. Doesn't matter how good it is.
